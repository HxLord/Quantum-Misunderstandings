# HELIOS GUARD - Technical Documentation

## Project Description

**Helios Guard** - это MVP-система защиты спутников от солнечных бурь и космической погоды. Система реализует полный цикл обнаружения, анализа и защитого маневра.

### Назначение

Система предназначена для демонстрации концепции TinyML-based защита спутников:
- Мониторинг солнечной активности в реальном времени
- Раннее обнаружение опасных событий (вспышки Х-Класса)
- Расчет времени прибытия частиц (ТоА)
- Автоматическое выполнение защитных команд

### Компоненты

1. **SensorUnit** (`SensorUnit` class)
- Генерирует симулированные данные солнечной активности
- Классы вспышек: A, B, C, M, X
- Скорость СМE: 300-3000 km/s

2. **TinyMl_Trigger** (`TinyML_Trigger` class)
- Пороговый детектор (Threshold potential: 1e-4 W/m²)
- Режимы: Sleep (0.01w) / Active (2.0w)
- Активирует AI при превышении порога

3. **MissionAI** (`MissionAI` class)
- Анализ угрозы
- Расчет времени прибытия: T = D / V
- Генерация защитных команд

## Physics

### Расчет времени прибытия (ТоА)

```
T = ( D / V ) - ( D / c )

Где:
    D - расстояние до солнца = 150,000,000 км
    V - скорость солнечных частиц (км/с)
    T - время прибытия (с)
    c - Скорость света в вакууме = 300,000 км/с 
```
### Классификация вспышек

| Класс | Поток (W/m²) | Опастность |
|-------|--------------|------------|
| A | < 1e-7 | Минимальная |
| B | 1e-7 - 1e-6 | Низкая |
| C | 1e-6 - 1e-5 | Умеренная |
| M | 1e-5 - 1e-4 | Высокая |
| X | > 1e-4 | Критическая |

# Техническая документация проекта на базе: Cesium sandcastle

## 1. Архитектура системы
Система построена на событийно-ориентированной модели. Основной цикл (Polling) запрашивает данные у внешнего API, передает их в аналитический модуль, который обновляет состояние объектов 3D-сцены и UI-интерфейса.

### 1.1 Модуль получения данных (Data Acquisition)
Используется эндпоинт `https://api.nasa.gov/DONKI/FLR`. 
* **Интервал опроса:** 20 секунд (оптимизировано под лимиты `DEMO_KEY`).
* **Обработка:** Извлечение параметра `classType` (например, "X1.2").

### 1.2 Аналитический модуль (Decision Logic)
Логика принятия решений базируется на парсинге первого символа типа вспышки:
```javascript
let threatLevel = type.charAt(0);
// Логика switch-case определяет состояние спутника
```
## 2. Визуальные компоненты

### 2.1 Спутник (Entity)

Спутник представлен объектом `Cesium.Entity` на геостационарной орбите ($h \approx 35,786$ км). Цвет точки (`point.color`) динамически меняется в зависимости от уровня радиационной угрозы.

### 2.2 Канал связи (Comms Link)

Реализован через `PolylineDashMaterialProperty`.DashLength и Color модулируются аналитическим блоком для имитации помех в ионосфере при высокой солнечной активности.

## 3. Интерфейс (UI/HUD)

Интерфейс реализован как HTML-overlay поверх Canvas.Критическое состояние: При обнаружении вспышки класса X, CSS-анимация `@keyframes alarm-pulse` активирует полноэкранную световую индикацию.Логирование: Динамический блок сообщений отображает историю принятых бортовым компьютером решений.

## 4. Математическая модель затухания (Перспектива)

### 4.1 Расчет потерь (Loss Calculation)

Для будущих обновлений планируется внедрение расчета потерь сигнала в зависимости от индекса солнечной активности по формуле:
$$L_{ion} = f(Flare_{class}, \theta)$$
Где $L_{ion}$ — потери в ионосфере, а $\theta$ — угол возвышения спутника.

# Техническая документация проекта на базе: Python(pygame+matplotlib)

### 1. Теоретическая база
Симуляция базируется на исследовании влияния высокоэнергетических частиц на космическую электронику. Модель учитывает три критических фактора:
* **Классификация вспышек**: Частицы делятся на классы (A, B, C, M, X) в зависимости от мощности потока ($W/m^2$).
* **Деградация компонентов**: Прохождение частиц через аппарат генерирует заряды, приводящие к «залипанию битов» и коротким замыканиям.
* **Защитные маневры**: Перевод систем в безопасный режим для минимизации площади поражения и риска "защелкивания".

### 2. Архитектура кода
Программа построена на объектно-ориентированном подходе:
* **`FlareParticle`**: Отвечает за генерацию векторов движения, расчет урона и визуализацию "шлейфа" частицы.
* **`Satellite`**: Моделирует поведение орбитального узла, его тепловой баланс и логику переключения режимов.
* **`reset_simulation()`**: Обеспечивает полную очистку оперативной памяти от старых объектов и сброс графического интерфейса.

### 3. Обработка ошибок и Графика
Для мониторинга используется интеграция `Matplotlib` в событийный цикл `Pygame`.
* **Логарифмическая шкала**: Позволяет визуализировать как слабый фоновый шум ($10^{-9}$), так и экстремальные вспышки ($10^{-4}$) на одном экране.
* **Валидация данных**: Внедрена система проверки индексов при отрисовке вертикальных линий защиты, что предотвращает `IndexError` при динамическом обновлении данных.

### 4. Математические модели
Урон рассчитывается как функция от класса частицы и текущего режима спутника:
$$Damage_{result} = Damage_{base} \times Multiplier_{mode}$$
* Где $Multiplier_{ACTIVE} = 1.2$, а $Multiplier_{SHIELD} = 0.05$ (снижение воздействия на 95%).

# Technical Documentation: Helios Guard 1D-CNN

## Архитектура нейросети
Модель спроектирована как одномерная сверточная сеть (1D-CNN) для анализа временных рядов. Выбор пал на CNN вместо LSTM из-за жестких ограничений по памяти и энергопотреблению на борту малых спутников.

### Спецификация слоев:

| Слой | Тип | Параметры | Обоснование |
| :--- | :--- | :--- | :--- |
| **Input** | Input | (60, 2) | Окно в 60 секунд, 2 канала (X-ray, Proton) |
| **Conv1D** | Convolution | 16 filters, Kernel 3 | Извлечение локальных признаков (градиенты, скачки) |
| **MaxPool** | Pooling | Pool size 2 | Снижение размерности, фильтрация шума датчиков |
| **Dense** | Fully Connected | 16 units, ReLU | Интерпретация извлеченных признаков |
| **Output** | Dense | 1 unit, Sigmoid | Расчет вероятности наступления события P(flare) |

## Пайплайн обработки данных

1. **Log-scaling:** Данные логарифмируются для линеаризации экспоненциального роста солнечного потока.
2. **Sliding Window:** Создание перекрывающихся окон данных (Stride=1) для непрерывного мониторинга.
3. **Quantization:** Пост-тренировочное квантование (INT8) через TFLite Converter для работы на микроконтроллерах без FPU.

## Стратегия защиты (Safe Mode)
При достижении порога `P(flare) > 0.8` система инициирует:
- **Mechanical:** Разворот аппарата защищенной стороной (Attitude Control).
- **Electrical:** Перевод второстепенных шин питания в режим Low Power.
- **Data:** Приостановка записи в энергозависимую память (SDRAM).

## Интеграция с NASA API
Скрипт `donki_api.py` обеспечивает возможность дообучения модели на реальных исторических данных через NASA Space Weather Database Of Notifications, Knowledge, Information (DONKI).
